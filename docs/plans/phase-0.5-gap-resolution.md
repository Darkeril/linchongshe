## 一、设计稿缺陷（需用户 + 设计师确认）

### 1.1 虚拟货币体系统一
**背景**：设计稿中同时出现 现金 / 爪币 / 爱宠豆 / 积分 / 成长值 / 红包 / 优惠券，口径不统一。Checkout 页的"爱宠豆"是否等于 Wallet 页的"爪币"？与 VIP 成长值是否相关？

**决策**：✅ 已定

#### 1.1.1 体系构成（保留 4 类）
1. **爪币**（虚拟货币）— 唯一的虚拟货币，不可提现
2. **VIP 成长值** — 只升级 VIP / 解锁权益，不可消费、不可提现
3. **优惠券** — 仅限商城抵扣使用
4. **红包** — 促销赠送，本质为优惠券子类（先留入口，具体玩法后续定）

#### 1.1.2 命名与展示
- 设计稿中所有"爱宠豆"、"积分"统一改为"**爪币**"
- 设计稿所有地方统一按上述四类来表达

#### 1.1.3 获取方式（需要可扩展）
- **爪币**：签到、发帖、互动（点赞/评论/分享）、完成任务（运营后台配置的任务，可随时增删）
- **VIP 成长值**：VIP 行为产生（购买 VIP、续费、消费等，后续细化）
- **优惠券**：商城活动、运营发放、任务奖励
- **红包**：促销场景赠送（具体获取路径待定，先留设计入口）

> ⚠️ **架构约束**：获取方式**必须后台可配置**，不能硬编码。前端通过"我的任务"等页面拉取动态任务清单。

#### 1.1.4 消费 / 用途
| 类型 | 用途 | 能换现金？ |
|------|------|----------|
| 爪币 | 兑换实物、兑换优惠券 | ❌ 不能抵扣现金 |
| VIP 成长值 | 解锁 VIP 等级 / 权益 | ❌ 不可消费 |
| 优惠券 | 商城下单抵扣 | ✅（以券面值扣减订单） |
| 红包 | 本质为优惠券，规则同上 | ✅（同优惠券） |

#### 1.1.5 时效性
- **所有四类资产都按"可过期"设计**（字段层面支持有效期）
- 具体过期规则（多久过期、过期提醒策略）开发到对应页面时再定

#### 1.1.6 对前端实现的影响
- 爪币、VIP 成长值、优惠券、红包各自独立的数据模型和接口
- 所有资产展示必须带"有效期 / 过期时间"字段，即使当前 UI 不显示也要预留
- 任务系统前端做"拉取任务列表 → 展示 → 点击完成"的通用框架，不与具体任务耦合

### 1.3 登录态拦截策略
**背景**：点赞 / 评论 / 关注 / 收藏 / 私信 / 发布 / 购买 等交互在未登录态时的全局策略未定义。这是图文社区最关键的转化漏斗断点。

**决策**：✅ 已定

#### 1.3.1 浏览态策略：完全开放浏览（对标小红书）
- 未登录用户可以看：**首页、笔记详情、用户主页、搜索**
- 只在触发"写操作"时才拦截要求登录
- ⚠️ 后端需提供未登录态的推荐算法（冷启动）

**需要登录的行为清单**：
- 点赞 / 收藏 / 关注 / 评论
- 发布笔记 / 视频 / 话题
- 私信 / 聊天
- 查看"我的" Tab
- 电商下单 / 加购物车
- 订单相关全部
- 设置 / 个人资料相关

#### 1.3.2 拦截 UI：底部半屏登录面板（Sheet）
- 保持上下文不跳出（当前页面不离开）
- 面板内含手机号+验证码登录 / 第三方登录（微信、抖音、QQ 等，具体清单后续定）
- 面板可下滑关闭

#### 1.3.3 动作回放：登录成功后自动执行原动作
- 拦截时必须缓存原动作的目标（笔记 ID、用户 ID、评论内容等）
- 登录成功关闭面板后，自动执行该动作
- 需要封装为**通用动作队列**，任何控件都能用（实现细节：action payload 统一格式，dispatch 队列）

#### 1.3.4 登录态失效：Token 自动刷新为主，主动 + 懒检测为兜底
- **Token 双 token 方案**：accessToken（短期）+ refreshToken（长期）
- accessToken 临过期自动用 refreshToken 刷新，用户无感
- **尽量不要出现过期情况**
- 如果 refreshToken 也失效 → 才启用兜底：
  - App 启动时主动校验一次
  - 运行时接口调用被拦截时懒检测兜底
  - 失效时触发 1.3.2 的登录面板，保留当前页面和动作

#### 1.3.5 对前端实现的影响
- 需要一个全局的**登录拦截中间件**（如 `useAuthGuard` composable）
- 所有"写操作"按钮调用前先过中间件
- 中间件负责：判断登录态 → 未登录则缓存动作 + 弹面板 → 登录成功后回放
- 路由守卫：需要登录的页面（我的、设置、订单等）未登录直接重定向到 Splash 或首页 + 弹登录面板

### 2.2 API 规范整改
**背景**：大量写操作使用 GET（`followById`、`pinnedNote`、`deleteCommentById`、`loginOut`、`deleteAlbum` 等），敏感 token 经 query 传递（`getUserInfoByToken?accessToken=`、`refreshToken?refreshToken=`），存在 CSRF 与日志泄露风险。

**决策**：✅ 已定

#### 2.2.1 方案：选项 A — 后端全面整改（与 2.1 一起做）
- 所有 GET 写操作改为 POST / PUT / DELETE
- token 一律用 Header 传递（`Authorization: Bearer xxx`）
- refreshToken 改用 **HttpOnly Cookie** 传递（SameSite + Secure，跨域方案后端配合）

#### 2.2.2 时间策略
- **不阻塞前端**：后端重构排期由后端团队决定，前端不等
- 前端从**第一行代码**就按规范写，即使后端暂未整改也按规范方式写接口签名
- 前端阶段全用假数据开发，接入真接口时如果后端还没改，先报错暴露出来，推动后端改

#### 2.2.3 前端实现约定
- **请求封装层**（`src/api/request.ts`）从一开始就：
  - token 放 Header：`Authorization: Bearer ${accessToken}`
  - refreshToken 走 Cookie，前端不手动管（浏览器/小程序 storage 自动带）
  - 写操作一律用 POST / PUT / DELETE，不允许出现 GET + 写语义
- **拦截器**从一开始就处理：
  - 401 → 尝试用 refreshToken（通过 Cookie 自动带）刷新 → 成功就重放请求；失败触发全局登录拦截
  - accessToken 临过期主动刷新
- **请求方法枚举**：service 层定义接口时 method 明确，不混用

#### 2.2.4 前端产出 TODO（后端参考）
前端开发过程中发现的所有"后端当前是 GET 写操作 / token 走 query"的接口，**都会在 `aichongshe-app/docs/api-todo.md` 的"后端重构依赖"章节单独列出**：
- 当前错误签名
- 建议的规范签名
- 前端假定的新签名
- 后端重构状态

后端重构这些接口时按此清单推进。

#### 2.2.5 小程序端的 Cookie 特殊处理
⚠️ **注意**：微信/抖音小程序的 `wx.request` / `tt.request` 对 Cookie 支持和普通浏览器不同，HttpOnly Cookie 方案需要验证：
- H5 端：原生支持 HttpOnly Cookie
- App 端（UniApp 原生打包）：取决于 WebView 配置
- 小程序端：可能需要手动管理（通过 response header 拿到 Set-Cookie 并存 storage）
- 在实际对接真实接口时要验证此点，如小程序端不支持，需要折衷方案（如 refreshToken 也走 Header，但加强防护）
